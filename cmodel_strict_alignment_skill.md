# C-Model Strict Alignment Skill (C-Model 严格对齐与精确描述原则)

## 背景
在撰写硬件架构、算法分析或 RDO (Rate-Distortion Optimization) 相关的技术文档时，容易受到通用算法或概括性认知的影响，从而使用夸张或模糊的表述（例如：“在海量可能的运动矢量中搜索”、“通常会做 XXX 操作”）。这会导致文档与实际硬件和底层参考模型 (C-Model) 严重脱节。

## 核心准则 (The Golden Rule)
**任何技术文档的撰写，必须 100% 严格依照 C-Model 中的具体代码实现和实际物理情况进行描述，杜绝一切概括性、夸张性或想当然的模糊词汇。**

## 最佳实践规范
1. **摒弃夸张修辞，回归硬核数据：**
   - ❌ 错误表述：“在成千上万个海量运动矢量候选中进行筛选...”
   - ✅ 正确表述：“在基于 MVP 的有限网格（如 3x3 九点全搜）中进行筛选...”
   - **执行要求：** 必须明确指出具体的网格大小、候选队列深度（例如：最终送入 RDO 的是 3 组精英候选，而不是模糊的“几个候选”）。

2. **拒绝“通常”，追溯源码事实：**
   - 不允许使用“一般情况下”、“通常”、“可能”等含糊其辞的假设。
   - **执行要求：** 所有的算法逻辑（尤其是降维策略、边界处理）都必须能在 C-Model 代码中找到对应的宏定义或逻辑代码作为支撑。
   - *案例：* 在分析 CU64 这种超大块的处理时，不能靠猜，必须明确翻阅代码。例如 Inter 采用的是底层 32x32 结果的**均值推导 (Average)**（`rcMv[0]->m_iHor + rcMv[1]->m_iHor + ...`），而 Intra 采用的是在 `intra_mode.h` 中通过 `#define NO_CU_64x64 1` 强制开启的**硬性碎化并发与物理叠加**。二者截然不同，必须精准定调。

3. **强调时序与依赖 (Timing & Dependency)：**
   - 硬件和底层算法对时序极其敏感。描述过程时，必须准确反映操作的先后顺序和依赖关系。
   - **执行要求：** 如果一个大的操作（如得出 CU64 的代价）依赖于若干个子块操作（如 4 个 32x32），必须明确写出“**等待**所有子块全部完成”这样的时序依赖机制，而不能让人误解为“只要计算一部分就能提前结束”。

4. **强制图表标注 (Mandatory Image Captions)：**
   - 任何嵌入文档的架构图、流程图、数据流图或算法示意图，下方都必须配备说明性的图注（如：*图 X：[图片内容核心说明]*）。
   - **执行要求：** 绝不能有“孤立”的图片存在，图注内容必须准确解释图片在算法或物理硬件中对应的具体环节。并且如果图表存在可能的歧义（如把 Band ID 误认为像素），必须添加醒目的图例警告。

5. **交互式动态可视化 (Interactive Dynamic Visualization)：**
   - 硬件与算法中存在大量动态游走的过程（如滑动窗口寻优、像素级地毯式扫描等），使用静态截图往往会导致理解断层或歧义。
   - **执行要求：** 对于具有时间轴或空间游走特性的逻辑，优先采用 HTML/JS 构建轻量级的交互式动画（Interactive Animation），让读者通过“点击运行”直观感受数据或扫描框在物理空间中的变化。

6. **架构级执行时序对齐 (Architectural Execution Timeline Alignment)：**
   - 初学者极易根据高级语言的思想脑补执行步骤（例如：“先挑出几个带，然后再去图里数像素”），这会严重背离底层为了节省 Memory Bandwidth 而设计的“一次过检，全盘统计”的真实硬件时序。
   - **执行要求：** 必须在文档中显式地梳理并强调**真实的执行流与时序先后关系**。明确区分“统计阶段（摸底）”与“决策阶段（滑动窗口查表）”的先后顺序。

7. **RDO 的经济学解释 (ROI Economic Analogy for RDO)：**
   - 解释 RDO 抉择时，不能仅仅扔出 `Cost = D + λ*R` 这个枯燥的数学公式。这无法回答“为什么要选 Cost 最小的？”这种本质疑问。
   - **执行要求：** 必须使用“投资回报率 (ROI)”、“花最少的 bit 办最大的画质提升”等经济学常识进行类比，将抽象的代价计算转化为直观的经济账，解释为什么不选“人数最多”或“失真最大”的局部最优解。

8. **变量物理意义溯源 (Trace Physical Meaning of Variables)：**
   - 源码中的特定约束（如 EO 强制规定 Class 1/2 为正偏移，Class 3/4 为负偏移）绝非心血来潮，背后必有深层的物理原因。
   - **执行要求：** 禁止仅仅把代码规则翻译成人类文字，必须深挖其背后的**物理/图像学意义**（例如：为了消除量化带来的 Gibbs 现象/振铃效应导致的高频下冲与过冲），打通代码约束与现实世界的联系。

9. **CABAC 码率代价（Bincost）四步推演规范：**
   - 文档中评估单个语法元素（如 `sig_coeff_flag`）的 Fractional Bits 时，禁止只写「约 1.2 bits」这类粗糙估数。
   - **必须走完且数字自洽的四步链：** ① 语法元素 → ctxInc；② Context_Array + QP → m_ucState；③ Entropy_LUT 建表/寻址；④ 查表得最终 bit 数。
   - **禁止**用「Level=3 逐项累加 sig/gr1/gr2」替代上述四步；四步推演写清后，多候选 Level 的累加逻辑读者自然能推。
   - 数值必须与 C-Model / 规范表对齐；详见 RDOQ Dashboard 1.3 节实战块。

## 总结
作为 AI 编码助手，在面对极其严苛的底层硬件和芯片 C-Model 研发环境时，必须收起发散思维，化身为**最严谨的源码翻译官**。一切推论和图表，皆以 C-Model 代码事实为唯一准绳。
